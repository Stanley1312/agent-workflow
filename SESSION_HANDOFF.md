# AI Workflow Project — Session Handoff
> Updated: 2026-05-10 | Use this file to onboard a new Claude session

## What this project is
A **Spec-First, Wiki-Driven, Wave-Based** AI development workflow for Claude Code.
The goal: eliminate AI inconsistency and forgetfulness by making `.ai/` the single source of truth for all agents, skills, and workflow logic.

Future vision: build a CLI that ports `.ai/` to other providers (Gemini, etc.) — **not in scope yet**.

## The Golden Loop (5 steps)
Step 1: DISCOVERY (Strategist)
→ Adaptive interview (one question at a time, build on answers)
→ Scan codebase first if project already exists
→ Confirm synthesis → generate CLAUDE.md into root, delete src/ placeholder
Step 2: TECHNICAL DESIGN (Architect)
→ Pre-SPEC ritual: read wiki + WebSearch/WebFetch current best practices
→ Write SPEC → HARD GATE: user must type "approve"
→ After approved: create STATE.md first (checkpoint: SPEC APPROVED), then PLAN.md
Step 3: TEST GENERATION (Tester) ← per wave
→ Tests organized by business domain, not wave name
→ File: src/[domain]/[domain].test.[ext]
→ Confirm all RED before handoff
Step 4: IMPLEMENTATION (Implementor) ← per wave
→ Green phase simple→complex
→ Retry limit 3 → escalate Architect (with full report) → escalate user
→ Update STATE.md after full wave (not per test)
Repeat Step 3+4 per wave
Step 5: VERIFICATION + INGESTION (Verifier → Architect)
→ V1 tests, V2 linter, V3 SPEC coverage, V4 UI (mandatory if HTML/JSX/Vue/Svelte), V5 security
→ Outcomes: ✅ PASS / ⚠️ WARN / ❌ FAIL
→ Bug routing inline — layer cascade: SPEC → PLAN → Test → Code
→ Architect: ingestion + gitnexus + archive

## Key design decisions (all finalized)

**Agent format**
Valid Claude Code subagent: `name`, `description`, `tools`, `model` only.
`description` = routing trigger — written like a rule, not a label.
Files live in `.claude/agents/`.

**SPEC Approval Gate**
Agent asks: "SPEC is ready. Do you approve? Type 'approve' to continue..."
User types "approve" / "ok" / "yes" → agent writes APPROVED + date into SPEC.md.
Never proceeds without this.

**STATE.md creation**
Created by Architect immediately after SPEC approved, before PLAN.
First checkpoint: `SPEC APPROVED — [date]`. Mandatory.

**Verifier outcomes**
- ✅ PASS: no FAIL in any step
- ⚠️ WARN: V2 warnings only — log in STATE.md, proceed
- ❌ FAIL: any step fails — stop, route to correct agent
V4: mandatory if project has `templates/`, `*.html`, `*.jsx`, `*.vue`, `*.svelte`. Cannot self-declare N/A.

**Implementor retry limit**
3 attempts max → escalate Architect with full report (test name, 3 attempts, exact errors).
Architect reviews plan, uses WebSearch/WebFetch, gives specific guidance.
Still stuck → escalate user. No infinite loops.

**Wave naming**
Domain-based: `Wave 1: Authentication`, not `Wave 1`.
Test files: `src/auth/auth.test.ts`, not `wave1.test.ts`.

**Bug routing**
Inline in `run.md` — not in footnotes.
Layer cascade: SPEC → PLAN → Test → Code (always top-down).
Layer dispute: 2 failed attempts → escalate user, do not self-loop.

**Web search**
Built-in: `WebSearch` + `WebFetch` (no MCP needed).
Used by: Architect (Pre-SPEC ritual), Debugger (Phase 4 external research).

**REQUIREMENTS.md format**
Each item: What / Why / Done when. No vague placeholders.
Priority: P0 / P1 / P2.

**ROADMAP.md format**
Macro only: milestone name, goal, status, feature titles.
No implementation details, no task lists.

**discuss.md**
General discussion only — probing, synthesis, challenge.
Discovery tree lives in `init.md`.

**Skill file structure**
`workflow/` = 3 files: `SKILL.md` (entry), `init.md`, `run.md`.
Split only when genuinely reused — no structural over-engineering.
