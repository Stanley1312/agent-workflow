# AI Workflow Project — Session Handoff
> Created: 2025 | Use this file to onboard a new Claude session

## What this project is
A **Spec-First, Wiki-Driven, Wave-Based** AI development workflow for Claude Code.
The goal: eliminate AI inconsistency and forgetfulness by making `.ai/` the single source of truth for all agents, skills, and workflow logic.

Future vision: build a CLI that ports `.ai/` to other providers (Gemini, etc.) — **not in scope yet**.

## The 6 Agents

| Agent | Model | Role | Key Skills |
|-------|-------|------|-----------|
| Strategist | opus | Discovery, Requirements, CLAUDE.md Part 1 | discuss.md |
| Architect | opus | SPEC, PLAN, Ingestion, Queue management | gitnexus, wiki_agent |
| Tester | sonnet | Write test cases (Red phase) | gitnexus |
| Implementor | haiku | Code to pass tests (Green + Refactor) | gitnexus |
| Verifier | sonnet | Quality gate: SPEC coverage + tools + trigger Tester re-run | playwright |
| Debugger | opus | Root cause investigation, recommend re-run scope | gitnexus, wiki_agent |

## The Golden Loop (5 steps)

```
Step 1: DISCOVERY (Strategist + discuss skill)
  → Interview user, confirm synthesis, update REQUIREMENTS.md

Step 2: TECHNICAL DESIGN (Architect)
  → Read wiki, use GitNexus, write SPEC → user approval → write PLAN
  → HARD GATE: no proceed without explicit SPEC approval

Step 3: TEST GENERATION (Tester) ← per wave
  → Write failing tests based on SPEC acceptance criteria

Step 4: IMPLEMENTATION (Implementor) ← per wave
  → Green phase + Refactor, update STATE.md checkpoints

  Repeat Step 3+4 per wave

Step 5: VERIFICATION + INGESTION (Verifier → Architect)
  → Verifier: test suite + linter + SPEC coverage + playwright + security
  → Architect: wiki_agent Ingest + gitnexus re-index + archive active/
```

## Key design decisions (all finalized)

**PLAN.md wave dependencies**
Each wave declares `Dependencies: [Wave X]` explicitly.
Re-run scope after bug = only affected wave + waves that depend on it.

**Bug fix flow**
```
Verifier/user finds bug
  → /debug → Debugger investigates (wiki + gitnexus + history)
  → Debugger outputs: root layer + affected waves + recommended action
  → Architect approves recommendation

If current feature bug:
  → Reflect to correct layer (SPEC/PLAN/Test/Code)
  → Re-run affected waves only

If legacy feature bug:
  → STATE.md → PAUSED
  → active/current/ → active/paused/
  → New bug fix task in active/current/
  → Full 5-step workflow
  → /workflow resume after bug fix done
```

**Chain of Truth**
`SPEC → PLAN → Tests → Code`
Never patch code without updating the layer above first.

**active/ folder states**
```
current/  ← one task at a time
paused/   ← only during legacy bug interrupt
```
No `queued/` folder — REQUIREMENTS.md is the queue.

**Skills vs Commands**
- `.claude/skills/` = entry points (thin wrappers, create slash commands)
- `.ai/skills/` = source logic (where agents read from)
- `.claude/full_workflow.md` = documentation only, @imported in CLAUDE.md

**discuss.md**
Merged `discuss` command + `discussion_protocol` rules into one file.
Strategist loads it. User can invoke /discuss independently.

**full_workflow.md**
Documentation only — not a skill, not executable.
Lives in `.claude/`, @imported into CLAUDE.md Part 2.

**wiki_agent.md**
Wraps user's `llm_coding_wiki.md` logic into a skill.
Has 4 workflows: Init / Ingest / Query / Lint.
Called by Architect (Ingest after every task, Query before SPEC).
Called by Debugger (Query during investigation).

**Model selection rationale**
- opus: tasks requiring deep reasoning (Strategist, Architect, Debugger)
- sonnet: tasks requiring understanding but not max reasoning (Tester, Verifier)
- haiku: execution-only tasks (Implementor — just make tests pass)

**llm-wiki/ is gitignored**
Knowledge base is local, not committed to repo.

**Skills format**
YAML frontmatter with `name`, `description`, compatible agents.
Claude Code: `.claude/skills/[name]/SKILL.md` creates `/name` slash command.
Model invocation is ON for all skills (Claude can auto-invoke when relevant).
